## Summary

Add an OAuth-based account-add flow for Claude Max / Pro / Team subscriptions so operators can link a Claude account in one click instead of manually extracting tokens from a Claude Code session and pasting them into the dashboard.

Operators now click **Add Claude account via OAuth** in the dashboard, copy a generated authorization URL, complete consent on `claude.ai`, and paste the resulting code back into the dialog. `codex-lb` performs the authorization-code exchange (with PKCE), parses the `id_token` for identity claims, and persists the new account through the same encryption + insert path used by the existing manual-paste endpoint.

## Why this is separate from `add-claude-oauth-pool`

`add-claude-oauth-pool` (#1) explicitly deferred PKCE-based authorization code flow to a follow-up. This change is that follow-up. Existing endpoints are unchanged; manual-paste (`POST /api/claude/accounts`) is kept as a fallback.

## Endpoint contract

| Method | Path | Body | Response |
|--------|------|------|----------|
| POST | `/api/claude/oauth/start` | `{}` | `{flowId, authorizationUrl, stateToken, expiresInSeconds, callbackInstructions, redirectUri}` |
| GET  | `/api/claude/oauth/status?flowId=...` | — | `{flowId, status, errorMessage?, errorCode?, accountId?, startedAt, finishedAt?}` |
| POST | `/api/claude/oauth/callback` | `{flowId, code, state}` | `{status: "success", account: ClaudeAccountResponse}` |

Error envelope matches project standard: `{"error": {"code": "<flow-level error_code>", "message": "..."}}`. `error_code` values: `flow_not_found` (404), `flow_expired` (410), `flow_not_pending` (409), `state_mismatch` (400), `invalid_grant` (502), `anthropic_unreachable` (502), `id_token_missing` (400), `id_token_malformed` (400), `id_token_claims_incomplete` (400), `account_already_exists` (409).

`state_token` is exposed in `/start` response (under `stateToken`) so the dashboard dialog can submit it unchanged on `/callback`. The dashboard session is the trust boundary; `/status` continues to NOT echo the state token.

## Security

- PKCE S256 only (RFC 7636); verifier never leaves the server.
- CSRF state token (256-bit) via `secrets.compare_digest`; mismatch is a 400.
- Tokens encrypted via existing `app/core/crypto.py` envelope only. No new crypto primitives.
- Single in-flight flow per process; new `/start` supersedes any pending flow.
- Lazy TTL expiry (no background sweeper); default 600s.
- Plaintext tokens, codes, state, and PKCE verifiers MUST NOT appear in logs (regression test in `test_claude_oauth_service.py`).
- No edits to `app/modules/proxy/service.py` (ADR-0001 preserved).

## Known limitations / out of scope

- **Multi-replica state is process-local.** State machine in-memory; if `/start` and `/callback` land on different replicas the callback returns 404 `flow_not_found`. Matches the existing `app/modules/oauth/` caveat. Documented in `context.md`; sticky-session routing or shared state (Redis) is a separate follow-up.
- **`id_token` signature is not verified.** Matches existing project convention; documented in `context.md` and `design.md`.
- **No device-flow attempt against Anthropic.** No public endpoint documented at design time.
